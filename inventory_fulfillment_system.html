Inventory & Fulfillment System

Excel execution layer · SQL-ready data model · Built for high-volume trading card operations (TCGplayer ecosystem)
Data Modeling CSV Ingestion Operational Systems Performance Engineering Excel Dynamic Arrays SQL Migration Path

What it is

A proprietary inventory management + order fulfillment system built for large-scale trading card singles sales. It links scan-level item data (CSV imports) to physical storage locations (box/segment/batch), enabling fast retrieval without requiring rigid physical alphabetization forever.

Design goal: treat physical inventory as a queryable dataset — stable IDs, normalized attributes, and a location abstraction — so storage can be flexible while picking stays deterministic.

Why this matters (real operational constraints)

  • High SKU counts and frequent replenishment make constant re-sorting costly and risky (handling/damage).
  • Orders often require many one-off picks; retrieval speed is the bottleneck, not listing.
  • Inventory is physically rebalanced over time; the system must support fast remapping without rework.
  • Excel must remain responsive at scale → formulas and workflows must be performance-aware.

Architecture (SQL-ready by design)

Implemented in Excel for iteration speed, but modeled with relational patterns so the migration to SQL is mechanical: tables map 1:1, lookups become joins, and CSV ingestion becomes ETL.

Key design choice: separate identity (what the card is) from location (where it lives), using stable keys.

Core entities (high-level, proprietary details omitted):

  • Inventory fact table: one row per inventory instance with a surrogate row_id, attributes (name/set/condition/print), and a batch_id foreign key.
  • Batch/location dimension: maps batch_id → physical box_id + segment and a human-readable label for fast navigation.
  • Staging/pick table: ephemeral order paste + parsing + status flags (picked / depleted) to keep fulfillment fast and Excel lightweight.

Pipeline: from scan exports to pickable inventory

Ingestion is CSV-driven to stay tool-agnostic. Scans/listing exports from tools in the trading card ecosystem (including ManaBox and TCGplayer workflows) are normalized into a consistent schema.

  • Standardized headers and types for later DBMS compatibility.
  • Generated stable surrogate keys (row_id) to support deterministic updates and remaps.
  • Batch assignment ties each import chunk to a physical location without enforcing a single “correct” storage strategy.

Picking workflow (built for throughput)

The fulfillment workflow is intentionally optimized for a single-operator, high-frequency environment: quick lookup → pull → mark status → batch-deplete later to avoid slow, repetitive edits during packing.

  • Locator search: query by name and return candidate inventory rows with batch_id and physical location.
  • Stable reference: copy row_id during picking to ensure the correct inventory record is updated later.
  • Staged depletion: mark picked immediately; mark depleted after bulk inventory updates.
Operational optimization: designed to reduce physical reshuffling and minimize time spent editing inventory during order packing.

Performance engineering (Excel constraints)

The system was iterated to stay fast under growing row counts by avoiding patterns that force workbook-wide recalculation (e.g., ledger-style per-row depletion math across large logs).

  • Kept “state” tables simple and durable; kept “workflow” tables ephemeral.
  • Reduced dynamic-array fanout where it impacts save/recalc performance.
  • Used keys and dimensions to reduce repeated expensive lookups.

Skills demonstrated

  • Data modeling: normalized schema patterns, surrogate keys, fact/dimension separation, migration planning.
  • ETL / ingestion: CSV normalization, schema hygiene, tool-agnostic import design.
  • Systems design: decoupling identity from location; handling re-slotting/remaps without data churn.
  • Excel engineering: Tables + structured references, dynamic arrays (LET/FILTER/XLOOKUP), error handling, performance tradeoffs.
  • Operational rigor: workflow design for high-throughput fulfillment and low handling risk.

Roadmap (when scale requires a DBMS)

Excel is the current execution layer by choice; the underlying model is intentionally SQL-ready. When volume/throughput crosses the threshold for indexing, concurrency, and transactional updates, the system ports cleanly to a relational database.

  • Excel tables → SQL tables (same entities and keys)
  • Lookups → JOINs
  • CSV ingestion → ETL jobs
  • Staging workflows → transactional tables / queues
